# Neon Platform DNA

## What This App Is
Neon is a multi-tenant student management platform for tuition centres, with institution-scoped operations for admins, headmasters, tutors, and students. It combines operational workflows (provisioning, cohorts, attendance, timetables, materials, quizzes, community, notifications) with billing and platform revenue tracking.

Core goals:
- Keep centres operating with weak connectivity through local-first patterns.
- Enforce tenant and role boundaries by default.
- Support an admin-led onboarding and growth model with tiered plans.
- Evolve into a full platform-fee collection flow (headmaster to platform owner).

## Architecture Overview

### 1) Frontend Architecture
- Framework: Next.js App Router (`src/app`), React 19.
- UI stack: Tailwind CSS v4 + shadcn components.
- Data fetching: React Query for client read models.
- Rendering mix:
  - Server components/pages for privileged admin data fetches.
  - Client components for interactive dashboard workflows.

### 2) Backend Architecture
- Backend-as-a-service: Supabase (Postgres, Auth, RLS, Realtime).
- Access modes:
  - User-scoped clients (`createClient`) for normal app operations.
  - Service-role client (`createServiceRoleClient`) for controlled admin/server actions.
- Write path style:
  - Next.js Server Actions for most mutations.
  - Realtime listeners in selected admin/dash flows.

### 3) Data / Tenancy Architecture
- Primary tenant key: `institution_id`.
- Key domain tables:
  - Identity: `profiles`, `institutions`
  - Academics: `cohorts`, `materials`, `quizzes`, `quiz_attempts`
  - Operations: `attendance`, `timetables`, `forum_posts`, `forum_comments`
  - Finance: `cash_payments`, `financial_reports`, `platform_invoices`, `platform_payment_transactions`
  - System: `notifications`, `leads`
- Enforcement:
  - Postgres RLS + helper functions (`get_auth_institution_id`, role helper functions).
  - Role checks in server actions.

### 4) Offline-First Architecture
- Local DB: Dexie (`src/lib/db.ts`).
- Current offline-enabled domains:
  - Cohorts, timetable, forum activity, attendance drafts.
- Current pattern:
  - Read fallback to local cache where implemented.
  - Limited queued writes (not system-wide yet).

### 5) Billing and Platform Payments Architecture
- Tier logic centralized in `src/lib/billing.ts`.
- Current report generation:
  - Monthly snapshots generated by admin action (`generateMonthlyReport`).
  - Idempotent upsert for report month/institution.
- Platform fee collection scaffold:
  - `platform_invoices` + `platform_payment_transactions`.
  - DPOPay contracts in `src/lib/platform-payments.ts`.
  - Placeholder webhook at `src/app/api/payments/dpopay/webhook/route.ts`.

## Module-by-Module Architecture

### Public Site and Growth
- Landing page and value messaging in `src/app/page.tsx`.
- Lead capture routes/actions for admin conversion.
- Architecture: static/SSR hybrid content with simple server action ingestion.
- Status: Active.

### Auth and Identity
- Supabase Auth + profile record linkage (including trigger support in SQL bootstrap).
- Roles: `admin`, `headmaster`, `tutor`, `student`.
- Architecture: auth session cookies + profile role checks + route/layout guards.
- Status: Active.

### Institution Academic Operations
- Cohorts/materials/quizzes/timetable/attendance/community.
- Architecture: role-scoped reads/writes via Supabase, React Query caching, selective local storage.
- Status: Active (with offline parity pending in some modules).

### Notifications
- In-app notifications stored in `notifications`.
- Reminder generation action exists for fee reminders.
- Architecture: table-driven event messages + realtime client updates in shell areas.
- Status: Active.

### Admin Command Center
- Institution management, lead conversion, tier changes, trial management, reporting views.
- Architecture: service-role backed server actions with admin-only guards.
- Status: Active.

### Billing Core
- Tier + per-student billing formula managed in shared lib.
- Financial report snapshots stored monthly.
- Architecture: deterministic pricing helper + admin-triggered reporting action.
- Status: Active (manual trigger path).

### Platform Fee Collection (Headmaster -> Platform Owner)
- UI and schema scaffolding for platform invoices/transactions is in place.
- DPOPay adapter contract and webhook endpoint are scaffolded but intentionally non-live.
- Architecture target:
  - Invoice creation -> checkout init -> provider callback -> reconciliation -> final status.
- Status: Pending activation.

## Operational Readiness Notes

### Production Environment Requirements
- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- `NEXT_PUBLIC_APP_URL` (recommended for metadata and callback URL consistency)

### Security Posture
- Keep `SUPABASE_SERVICE_ROLE_KEY` server-only.
- Use strict role checks on every service-role mutation path.
- Enforce ownership validation for foreign IDs in service-role writes.
- Validate webhook signatures before any payment state transitions.

### Reliability Priorities (Next Steps)
1. Enable scheduled monthly invoice generation.
2. Implement DPOPay live adapter + signed webhook verification.
3. Add idempotent transaction reconciliation and retry policies.
4. Expand offline sync coverage and conflict handling.
